### [2026-02-21] ファイル名の英語タイトル部分にアンダースコア不要

**状況:** 文書ファイルのリネーム時

**原因:** タイトル部分もアンダースコア区切りにすると想定していた

**教訓:** 命名規則ではアンダースコアは Document ID とタイトルの「区切り」のみに使用。タイトル内はスペースのまま
**他プロジェクトへの応用可能性:** ❌

### [2026-02-21] 文書IDは番号だけでなく完全な形式を使用する

**状況:** DOC_REGISTRY のキーを `'3626'` としていた
**エラー内容:** ベトナムの文書IDは `{番号}-{種別}-{発行者}` が正式な形式。番号だけだと異なる発行者間で重複の可能性がある
**原因:** 文書IDの構造を正しく理解していなかった
**解決策:** `'3626-QD-DHQGHN'` のようにフルIDを使用。YAML では公式表記 `3626/QĐ-ĐHQGHN` を使い、ファイル名・JS キーではサニタイズ版を使用
**教訓:** ベトナム公文書のIDは `{番号}/{種別}-{発行者}` が正式。ファイルシステムでは `/` → `-`、ダイアクリティカル除去したサニタイズ版を使用
**他プロジェクトへの応用可能性:** ✅ ベトナムの公文書を扱うすべてのシステムで適用可能

### [2026-02-21] Firebase Compat SDK を使用する（単一ファイル SPA の場合）

**状況:** Firebase SDK を `index.html` に追加する際
**エラー内容:** ES Module 版 Firebase SDK は `<script type="module">` が必要で、既存の IIFE スクリプトとスコープが分離される
**原因:** ES Module は独自スコープを持つため、`firebase` オブジェクトがグローバルに公開されない
**解決策:** Firebase Compat SDK（`firebase-app-compat.js` 等）を使用し、`firebase` をグローバルスコープに配置
**教訓:** 単一ファイル SPA（`<script>` タグ内に全 JS を記述）では、Compat SDK が最も統合しやすい
**他プロジェクトへの応用可能性:** ✅ ビルドステップなしの静的サイトで Firebase を使う場合全般

### [2026-02-23] CSSの変更スコープを限定する — グローバルなリストスタイル変更の副作用

**状況:** điểm項目（a), b) 等）の箇条書き表示を改善しようとした
**エラー内容:** `.prose-legal ol` の `margin-left` を `1.25rem` → `1.5rem` に変更し、`ol > li { margin: 0.6rem }` を追加。結果、文書全体の khoản（番号付きリスト）が過剰にインデントされた
**原因:** điểm のフォーマット問題を解決するために、関係ない `<ol>` のグローバルスタイルまで変更してしまった。特定の要素のスタイルを修正する際に、影響範囲の広いセレクタを変更してしまうアンチパターン
**解決策:** CSS変更を元に戻し、điểm のインデントは Markdown プリプロセッサ（NBSP挿入）で対応。CSSは元の値を維持
**教訓:** CSS修正時は影響範囲を最小限にする。特にグローバルセレクタ（`ol`, `li`）の margin/padding 変更は文書全体に波及する。特定要素の問題は、CSSよりもHTML前処理（プリプロセッサ）で対応するほうが副作用が少ない
**他プロジェクトへの応用可能性:** ✅ CSS変更全般に適用可能

### [2026-02-23] Markdownの暗黙変換を理解する — `1. Text` が `<ol>` になる問題

**状況:** 法令文書の khoản（項）は `1. Text` `2. Text` 形式。marked.js がこれを `<ol><li>` に変換し、ブラウザのリストスタイル（margin/padding）が適用されて過剰インデントが発生
**エラー内容:** `<ol>` の margin + `<p>` の text-indent で二重字下げ。CSS で margin を調整しても `text-indent` との組み合わせで常にバランスが崩れた
**原因:** Markdown パーサーの暗黙的な変換（`1. ` → `<ol>`）が法令文書のレイアウトに不適合。CSS で後から修正しようとすると、`breaks: true/false` の違いで `<li>` 内の `<p>` 有無が変わり、text-indent の適用も変化する連鎖的な副作用が発生
**解決策:** プリプロセッサで `1. ` → `1\. ` にエスケープし、`<ol>` 変換自体を防止。khoản は通常段落として表示され、text-indent のみが適用される
**教訓:**
- **CSS での後処理より、入力の前処理が確実。** Markdown パーサーの出力を CSS で修正すると、パーサー設定（breaks 等）の変更で連鎖的に壊れる
- **`breaks: true` は `<li>` 内に `<p>` を生成する。** これにより text-indent 等の段落スタイルがリスト内にも適用され始める（breaks: false では発生しない）
- **法令文書の番号付き項目は HTML ordered list として扱うべきではない。** `1.` は段落の一部であり、ブラウザの `<ol>` レンダリング（マーカー、インデント）は不適切
**他プロジェクトへの応用可能性:** ✅ Markdown → HTML 変換で特定の構文を抑制したい場合全般

### [2026-02-23] 変更は必ずデプロイ先で確認する — ローカルテストとデプロイの乖離

**状況:** index.html の変更をローカルで確認し「動作しています」と報告したが、ユーザーは GitHub Pages を見ており変更が反映されていなかった
**エラー内容:** push していない状態で「修正完了」と報告。ユーザーの閲覧環境と自分のテスト環境が異なることを考慮していなかった
**原因:** ユーザーがどの環境で確認しているかを確認せずに作業完了を報告した
**解決策:** 変更後は必ず git push → GitHub Actions デプロイ完了を確認してから報告する
**教訓:**
- **ユーザーの閲覧環境を最初に確認する**（ローカル or デプロイ先）
- **push → デプロイ完了 → ハードリロード確認** の順序を徹底
- **視覚的な目印**（一時的なボーダー色変更等）でデプロイ反映を即座に判別できるようにする手法は有効だった
**他プロジェクトへの応用可能性:** ✅ CI/CD パイプラインを使うプロジェクト全般

### [2026-02-23] インデント手法は CSS クラスに統一する — NBSP vs CSS

**状況:** khoản（1., 2.）は CSS `padding-left` でインデント、điểm（a), b)）は NBSP 文字でインデントと、2つの異なる手法を使用していた
**エラー内容:** khoản と điểm で視覚的なインデントの質が異なる（NBSP は文字幅依存で不揃い、CSS は正確）。処理方式の違いにより保守性も低い
**原因:** điểm は最初にプリプロセッサで NBSP 挿入する方式で実装し、khoản は後から CSS クラス方式で追加したため、手法が分離した
**解決策:** điểm も khoản と同じ方式に統一 — プリプロセッサで空行挿入→個別 `<p>` に分離→post-render で `class="diem"` 付与→CSS `padding-left` でインデント
**教訓:**
- **インデント手法は1つに統一する。** NBSP 文字は文字幅依存で不揃いになりやすく、CSS `padding-left` の方が正確で保守しやすい
- **同じ種類の問題（リスト項目のインデント）には同じ解決パターンを適用する。** 空行挿入→`<p>` 分離→CSS クラス→`padding-left` のパイプラインを統一的に使用
**他プロジェクトへの応用可能性:** ✅ Markdown→HTML 変換でインデントが必要な全プロジェクト

### [2026-02-23] HTML正規表現は `<p>` 直後の空白を考慮する

**状況:** marked.js 出力の `<p>` タグに `class="khoan"` を追加する正規表現 `/<p>(\d+)\. /g`
**エラー内容:** khoản 1 にのみクラスが付与されず、khoản 2, 3 には付与された。結果、同じ条文内で khoản のスタイルが不統一
**原因:** marked.js は `<p>` タグの直後に改行 `\n` を挿入することがある。`/<p>(\d+)/` は `<p>\n1. ` にマッチしない
**解決策:** `/<p>\s*(\d+)\. /g` に変更し、`<p>` と内容の間の空白を許容
**教訓:** HTML パーサー出力を正規表現で処理する際、タグと内容の間の空白（改行・スペース）を常に `\s*` で許容する。可能であれば DOM API（querySelector + textContent）で処理する方が信頼性が高い
**他プロジェクトへの応用可能性:** ✅ HTML 後処理全般

### [2026-02-23] text-indent は breaks:true 環境で副作用が大きい

**状況:** `.prose-legal p { text-indent: 1.5em; }` で段落の字下げを実装
**エラー内容:** `breaks: true` の marked.js では `<li>` 内に `<p>` が生成され、リスト内の段落にも text-indent が適用される。さらに `<ol>` の margin + text-indent で二重インデントが発生。ユーザーには「段落全体が字下げされている」と認識された
**原因:** `text-indent` は CSS の first-line 擬似要素的な効果だが、すべての `<p>` に適用されるため、意図しないコンテキスト（リスト内、div 内）でも字下げが発生。`div p` の例外ルール追加で対処しようとしたが、対処箇所が増え続けるモグラ叩きになった
**解決策:** `text-indent` を完全に削除。khoản の判定（`class="khoan"` 付与）が正しく機能するようになったため、段落の先頭字下げ自体が不要になった。khoản にはハンギングインデント（`text-indent: -1.5em; padding-left: 1.5em`）を使用
**教訓:**
- **`breaks: true` + `text-indent` は相性が悪い。** `<li>` 内に `<p>` が生成されるため、リスト内にも段落スタイルが波及する
- **CSS の除外ルール（`div p`, `p[align]`）で対処するアプローチは脆い。** 新しいコンテキストが増えるたびにルール追加が必要
- **入力（Markdown）側で解決する方がCSSの副作用連鎖を避けられる**
**他プロジェクトへの応用可能性:** ✅ marked.js の `breaks: true` を使う全プロジェクト

### [2026-02-23] khoản（項）と箇条書きを混同しない — ベトナム法令文書の構造区別

**状況:** ベトナム法令文書の条文（Điều）を Markdown に転記する際、番号付き項目（1., 2., 3.）のインデントレベルを判断する場面
**エラー内容:** khoản（項）をリスト項目として扱い、4スペースインデントで記述してしまった。結果、Markdown パーサーがインデントコードブロックとして解釈するリスクが発生し、条文の階層構造も不明瞭になった
**原因:** khoản（番号付き条項）と箇条書き（`- text`）の構造的な違いを認識していなかった。PDFでは khoản は見出しからわずかにインデントされて見えるが、これは本文テキストと同じ位置であり、リストのインデントではない
**解決策:** khoản は条文（Điều）の直接の構成要素であり、本文と同じレベル（インデントなし）で記述する。箇条書き（`- text`）は khoản 内のサブ項目であり、インデントが付く。この区別を QA ワークフロー 1.4.4 とチェックリスト 1.5 に追記
**教訓:**
- **khoản（1., 2., 3.）は条文の段落区分であり、リスト項目ではない。** インデントは不要
- **箇条書き（`- text`）は khoản 内のサブ項目。** こちらにはインデントが付く
- **PDFの視覚的インデントに惑わされない。** 本文テキストと同じ水平位置なら、それはインデントではなく本文レベル
**他プロジェクトへの応用可能性:** ✅ ベトナム法令文書を構造化する全プロジェクト

# Repo QA

Use this skill for end-to-end document QA in this repository.

## Values
- This repository is a document archive.
- Accuracy and readability matter more than speed.
- Do not loosen QA criteria to make a batch pass.
- If a check fails, go back to the documents and fix them when it is safe.
- If a required check cannot be passed safely, leave the item open and prepare a clear user report.

## Trigger Conditions
Use this skill when any of the following is true:

1. This is the first task performed in this repository during the current conversation.
2. The user asks for repository maintenance or document maintenance here.
3. The user asks for document QA, transcription QA, PDF↔MD checking, translation QA, batch QA, or PR re-checks here.

## First-Invocation Briefing
On the first invocation in the current conversation:

1. Send a short English message saying what you will inspect first.
2. Mention the expected flow briefly: inventory, normalization, checks, fixes, report/status updates, and PR creation if requested.
3. Then start execution.

Later invocations in the same conversation may skip this unless the scope changes materially.

## Load Order
1. Read `skills/repo-qa/references/vju-project.md`.
2. Read only the relevant sections of `docs/QA_CHECKLIST.md`.
3. Load the runtime child skill that matches the current agent:
   - Codex runtime: `skills/repo-qa/children/codex-qa/SKILL.md`
   - Gemini runtime: `skills/repo-qa/children/gemini-qa/SKILL.md`
4. If the current document issuer matches a generated issuer child skill, load that child skill too.
5. Inspect the current scripts, report files, and status file before making decisions.

## Current Child-Skill Set
- Runtime children:
  - `skills/repo-qa/children/codex-qa/SKILL.md`
  - `skills/repo-qa/children/gemini-qa/SKILL.md`
- Issuer children generated by the repository threshold rule:
  - `skills/repo-qa/children/issuer-vju/SKILL.md`
- No additional issuer or issuer×type child skill should be created unless the repository threshold is met.

## Core Rules
- Keep user-facing updates in Japanese.
- Keep repository-facing names, branch names, PR titles/bodies, and commit messages in English.
- Never add AI co-author trailers such as `Co-authored-by`.
- Prefer deterministic checks and scripts before model judgment.
- Treat `docs/qa_report_master.md` and `Tasks.md` as append-only operational history.
- Keep `tmp/qa_status.json` synchronized for every touched `doc_id`.
- Never add `last_updated` to transcription front matter.
- During long runs, briefly re-open this skill and the relevant checklist section about every 5 minutes or after each major phase boundary.

## Work Window
- `DEFAULT_WORK_WINDOW = 30` document sets.
- If the workflow uses a count-based pause boundary, use 30, not 3 or 12.
- For full-repository runs, passing 30 items is not a stop reason by itself. Continue automatically until an explicit stop condition is reached.

## Required Files
Before execution, confirm these paths exist or can be initialized:

- `docs/qa_report_master.md`
- `Tasks.md`
- `tmp/qa_status.json`
- `docs/QA_CHECKLIST.md`

If the repository does not contain the required `repo-qa` files or contracts, stop and report that blocker clearly. Do not switch to another repository.

## Workflow
Run the work in this order:

1. Precheck
2. Inventory
3. Required filename normalization
4. Missing transcription generation or recovery
5. Source fidelity checks
6. Script checks
7. Document fixes
8. QA re-check
9. Strict reviewer pass
10. Report update
11. Status update
12. PR creation when requested

## Inventory And Priority
- Detect document sets by `*_source.pdf` and `*_transcription*.md`.
- The goal is repository-wide coverage, not a small sample.
- A set remains in scope until it passes all required QA gates or is explicitly blocked.
- Apply this priority order:
  1. Required safe renames
  2. Documents issued within the last 3 months
  3. Unfinished capability-gated work
  4. Remaining QA work for not-yet-passed items
  5. If all items already passed, re-check the 30 oldest checked sets

## Checks And Fixes
- The QA checklist defines pass/fail criteria. Do not weaken it during execution.
- If a document fails a check and the issue is safely fixable, fix it and rerun the relevant checks.
- Do not change checklist criteria or downgrade a failure to make the run pass.
- If the issue cannot be fixed safely, keep the item open and report why.
- Preserve layout fidelity in official headers, centered blocks, appendices, form structures, and signature sections.
- Heading-count mismatches are only signals. Confirm semantic structure before editing.
- Do not treat placeholders, TODO text, or “translation will be provided later” notes as completed restoration.

## Count Reporting
- Every run must make it obvious how many document sets were:
  - in scope
  - checked
  - passed
  - fixed but still open
  - blocked
  - skipped
- If the target count is 0, do not accept that casually.
- Re-check whether inventory logic, target root, filters, and status file interpretation are correct.
- If the correct result is still 0, report that explicitly to the user as a real result.

## Strict Reviewer Pass
After the ordinary QA pass:

1. Review the changes as a strict reviewer.
2. Look for regressions, misleading report claims, formatting drift, layout damage, metadata mistakes, and claim-vs-diff mismatches.
3. If problems are found, go back and fix the documents, scripts, or reports.
4. Do not improve the checklist or reinterpret the result to hide a defect.

## Self-Check Gate
Before declaring completion or creating a PR:

1. Confirm `docs/qa_report_master.md` preserved prior history and appended the new run.
2. Confirm `Tasks.md` preserved prior history and appended the new run.
3. Confirm `tmp/qa_status.json` is updated for every touched `doc_id`.
4. Confirm no touched transcription file gained `last_updated`.
5. Confirm no placeholder-only content is described as restored content.
6. Confirm layout-sensitive regions remain layout-faithful.
7. Confirm report claims match the final diff.
8. Confirm commit messages are in English and contain no AI co-author trailer.

If any self-check item fails:
- do not declare completion
- do not relax the checklist
- return to the relevant fix step

## Stop Conditions
Stop only when one of these is true:

1. No more eligible non-blocked work remains under `repo-qa`.
2. Only blocked work remains.
3. A real runtime or tool limit prevents safe continuation.

These are not valid stop reasons:
- “3 items are done”
- “12 items are done”
- “30 items are done”
- “a first batch is done”
- “a summary has been written”
- “the workflow was demonstrated”

## Expected Deliverables
At the end of a run, leave:

- updated document files where safe fixes were applied
- an appended `docs/qa_report_master.md`
- an appended `Tasks.md`
- an updated `tmp/qa_status.json`
- a concise Japanese summary to the user
- a PR, if the user requested PR creation

## PR Rule
- Create a PR when the user asks for it.
- Do not merge at this stage unless the user separately asks for merge.
